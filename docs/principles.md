# Project Principles

These principles guide every design and engineering decision in Project Memory.

---

## Principle 001

If a feature doesn't help answer **"WHY"**, it doesn't belong in Project Memory.

---

## Principle 002

Users should never feel like they are using a RAG application.

They should feel like they are talking to their project's memory.

---

## Principle 003

Every answer must be backed by evidence.

No unsupported conclusions.

---

## Principle 004

Project Memory reconstructs decisions.

It does not simply retrieve documents.

---

## Principle 005

AI is an assistant for reasoning.

It should never invent project history.

---

## Principle 006

Timeline is more valuable than isolated documents.

Always connect related events whenever possible.

---

## Principle 007

Every new feature must solve a real engineering problem.

No AI features for the sake of AI.

---

## Principle 008

Build for software teams first.

Generalization can happen later.

---

## Principle 009

Prefer explainability over intelligence.

Users should always understand why an answer was generated.

---

## Principle 010

Keep the product simple.

Every feature should make Project Memory easier to use, not more complicated.

---

## Principle 011

Documentation should support development.

Development should never stop because of documentation.

---

## Principle 012

Project Memory stores knowledge, not documents.

---

## Principle 013

Build with free resources first. Paid services should be optional, not required.

---

## Principle 014

Do not introduce abstraction without responsibility.

Every layer should have a clear purpose.

---
